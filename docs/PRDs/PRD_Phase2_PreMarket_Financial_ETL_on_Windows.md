# PRD: Phase2_PreMarket_Financial_ETL_on_Windows

## 1. Context & Problem (业务背景与核心痛点)
ISSUE-1103. In Phase 1, we established the core event engine. To calculate dynamic financial metrics (PE-TTM, PB, ROE) in real-time on Linux during trading hours without blocking or OOMing the gateway, we need a lightweight pre-calculated factor dictionary. The source of truth lies in QMT's local binary `.DAT` files on the Windows node. 

The core problem is: `xtdata` locks the `data_dir` upon its first import inside a long-running FastAPI process, making it impossible to query the `Finance/` directory via the current `server.py` bridge. We must write an independent, short-lived Python batch script on the Windows node to extract and reduce this massive financial dataset into a tiny JSON artifact daily before the market opens.

## 2. Requirements & User Stories (需求定义)
- **Functional Requirements:**
  - Create a new batch script `windows_bridge/finance_batch_etl.py` (stored in the AMS repo but intended to run on the Windows host).
  - The script must explicitly set `xtdata.data_dir = 'C:/国金证券QMT交易端/datadir'` BEFORE calling any `xtdata.get_financial_data()` functions.
  - It must fetch the full A-share market stock list via `xtdata.get_stock_list_in_sector('沪深A股')`.
  - It must batch-process these stocks (fetching `Capital`, `Balance`, and `Income` tables).
  - It extracts:
    - Total Capital (`total_capital` from Capital table)
    - Total Equity (`tot_shrhldr_eqy_excl_min_int` from Balance table, fallback to `total_equity`)
    - Net Profit (`net_profit_excl_min_int_inc` from Income table)
  - It must output a flat JSON file mapping stock codes to these 3 factors, ensuring `NaN` or invalid data is handled safely (e.g., stored as `null` or omitted).
- **Non-Functional Requirements:**
  - Fast batching: Do not query 5000 stocks in a single API call if it causes crashes; chunk them if necessary.
  - Idempotency: Running it multiple times should simply overwrite the JSON.

## 3. Architecture & Technical Strategy (架构设计与技术路线)
- **Local Proxy Pattern (Linux side)**: The actual execution happens on Windows, but the code lives in Linux under `windows_bridge/`. 
- **ETL Transformation Logic**: 
  - `xtdata.get_financial_data` returns a deeply nested dict: `{"000001.SZ": {"Capital": [dict], "Balance": [dict], "Income": [dict]}}`.
  - The script will iterate through the stocks, extract the first (most recent) item in each table's list.
  - It outputs a flat dictionary to a local file on Windows (e.g., `C:\Users\Administrator\Desktop\AMS\fundamentals.json`), which the FastAPI server will serve via a new endpoint.
- **FastAPI Bridge Upgrade (`server.py`)**:
  - Add a new endpoint `GET /api/fundamentals` in the Windows `server.py` that simply reads and returns the `fundamentals.json` file generated by the ETL script.

## 4. Acceptance Criteria (BDD 黑盒验收标准)
- **Scenario 1: Generating the ETL JSON on Windows**
  - **Given** the QMT terminal has downloaded financial data to `C:\国金证券QMT交易端\datadir\Finance`
  - **When** `finance_batch_etl.py` is executed
  - **Then** a `fundamentals.json` file is created containing a flat mapping of stock codes to `total_capital`, `total_equity`, and `net_profit`.
- **Scenario 2: Serving the ETL artifact via the Gateway**
  - **Given** the `fundamentals.json` exists on the Windows node
  - **When** a client hits `GET /api/fundamentals` on the FastAPI bridge
  - **Then** the JSON payload is successfully returned in milliseconds.

## 5. Overall Test Strategy & Quality Goal (测试策略与质量目标)

- **Sanitization**: Ensure the test checks that Pandas/Numpy `NaN` values are properly converted to Python `None` to prevent JSON serialization crashes.
